# 经验沉淀规范（重要）

每次完成非平凡任务（新增题目、批量修改、Bug 修复等）后，**必须**执行经验沉淀。

## 8.1 何时触发

- 新增了 2 道以上题目
- 发现并修复了通用性问题（如公式渲染、截断行、编码问题）
- 建立了新的编写模式或约定
- 用户反馈了质量问题并修复后

## 8.2 沉淀什么

| 类别 | 示例 |
|------|------|
| **踩坑记录** | `$n$` 单字母变量在 GitHub 不渲染 |
| **格式约定** | 块级公式前后必须空行 |
| **流程改进** | 总纲不维护数字，避免过时 |
| **质量规则** | 禁止 `\begin{aligned}` 环境 |
| **工具配置** | 各 Agent 的规则文件位置 |

## 8.3 沉淀到哪里

### 规范文件体系

本项目使用「主源 + 合并版」策略管理规范文件：

| 文件 | 适用工具 | 角色 |
|------|---------|------|
| `.claude/docs/*.md` | Claude Code、Trae | **主源**（Claude Code 自动加载） |
| `.cursor/rules/project-rules.md` | Cursor | 合并完整版 |
| `.github/copilot-instructions.md` | Copilot、Codex | 合并完整版 |

### 同步策略

1. **修改规范时**：先修改 `.claude/docs/` 下对应的子文件（主源）
2. **然后合并**：将变更合并到 `.cursor/rules/project-rules.md` 和 `.github/copilot-instructions.md`
3. **合并版内容**：与拆分前 CLAUDE.md 内容一致，是所有子文件的合并
4. **三个文件一起提交**：确保主源和合并版始终同步

### 各子文件职责

| 子文件 | 内容 |
|--------|------|
| `workflow.md` | 面试题生成工作流、新增题目工作流、编号规则 |
| `frontmatter.md` | YAML Frontmatter 字段规范、知识点关系 |
| `answer-structure.md` | 五步回答结构（结论→原理→工程→踩坑→延伸） |
| `latex-math.md` | LaTeX 符号约定、行内/块级判定、GitHub 兼容性、单字母陷阱 |
| `file-naming.md` | 文件命名规范 |
| `quality.md` | 内容质量要求、截断行检查、Unicode 转义检查 |
| `git-conventions.md` | Git 提交规范、分支/PR 工作流 |
| `retrospective.md` | 经验沉淀规范（本文件） |
| `checklist.md` | 新增内容检查清单 |

## 8.4 沉淀原则

- **具体化**：写明"是什么、为什么、怎么做"，不写空泛原则
- **可执行**：其他 Agent 读到后能直接遵循，无需额外推断
- **去重**：先检查是否已有相关规则，避免重复
- **标注来源**：如果是从 Bug 修复中总结的，简要说明触发场景
