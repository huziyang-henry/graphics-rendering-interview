# Q01.10 法线矩阵为什么要用逆转置矩阵？直接用模型矩阵变换法线会有什么问题？

**难度：** 🟡 中级
**所属章节：** [渲染管线基础](./index.md)

---

## 🎯 结论

- 法线是方向向量（Direction Vector），只受旋转和均匀缩放的影响，不受平移和非均匀缩放的影响。而模型矩阵 $\mathbf{M}$ 中包含平移分量和非均匀缩放分量，直接用它变换法线会破坏法线与表面的垂直关系。
- 正确的法线变换矩阵是模型矩阵左上角 3x3 子矩阵的逆转置：$\mathbf{N} = (\mathbf{M}_{3\times3}^{-1})^{\top}$。这个矩阵能保证变换后的法线仍然垂直于变换后的表面。
- 仅当模型矩阵只包含旋转和均匀缩放时，$(\mathbf{M}^{-1})^{\top}$ 才与 $\mathbf{M}$ 成比例关系，此时可以直接用模型矩阵变换法线后归一化。但在存在非均匀缩放时，必须使用逆转置矩阵。

## 📐 原理解析

### 法线与切线的垂直关系

- 在模型空间中，表面某点的切线向量 $\mathbf{t}$ 和法线向量 $\mathbf{n}$ 满足垂直关系：$\mathbf{n} \cdot \mathbf{t} = 0$，即 $\mathbf{n}^{\top} \mathbf{t} = 0$。
- 经过模型矩阵 $\mathbf{M}$ 变换后，切线变为 $\mathbf{t}' = \mathbf{M} \mathbf{t}$。切线是位置/方向向量，直接用模型矩阵变换是正确的。
- 法线变换后应满足 $\mathbf{n}' \cdot \mathbf{t}' = 0$，即变换后的法线仍然垂直于变换后的切线。

### 逆转置矩阵的数学推导

- 将 $\mathbf{t}' = \mathbf{M} \mathbf{t}$ 代入垂直条件：

$$\mathbf{n}'^{\top} \cdot \mathbf{M} \mathbf{t} = 0$$

- 我们知道 $\mathbf{n}^{\top} \mathbf{t} = 0$，要使上式对所有 $\mathbf{t}$ 成立，需要：

$$\mathbf{n}'^{\top} \mathbf{M} = \mathbf{n}^{\top}$$

- 两边取转置：

$$\mathbf{M}^{\top} \mathbf{n}' = \mathbf{n}$$

- 解出 $\mathbf{n}'$：

$$\mathbf{n}' = (\mathbf{M}^{\top})^{-1} \mathbf{n} = (\mathbf{M}^{-1})^{\top} \mathbf{n}$$

- 因此，法线变换矩阵是模型矩阵的逆转置矩阵。注意这里使用的是 $\mathbf{M}$ 的 3x3 子矩阵（去掉平移分量），因为法线是方向向量，不受平移影响。

### 均匀缩放的特殊情况

- 当 $\mathbf{M} = s \cdot \mathbf{R}$（均匀缩放 *s* 乘以旋转矩阵 $\mathbf{R}$）时：

$$\mathbf{M}^{-1} = \frac{1}{s} \mathbf{R}^{\top}, \quad (\mathbf{M}^{-1})^{\top} = \frac{1}{s} \mathbf{R}$$

- 此时 $(\mathbf{M}^{-1})^{\top} = \frac{1}{s^2} \mathbf{M}$，即逆转置矩阵与原矩阵成比例。因此可以直接用 $\mathbf{M}$ 变换法线，然后归一化即可得到正确结果。
- 但在非均匀缩放（如 $s_x \neq s_y$）时，这个比例关系不成立，必须使用完整的逆转置矩阵。

### 非均匀缩放导致法线偏移的几何直觉

- 想象一个单位正方体在 X 方向被拉伸为 2 倍。原来顶面的法线是 $(0, 1, 0)$，拉伸后顶面变为矩形，但法线方向不变，仍然是 $(0, 1, 0)$。
- 如果直接用模型矩阵变换，法线会变为 $(0, 2, 0)$，归一化后虽然方向正确，但这只是巧合——对于斜面法线，非均匀缩放会导致法线方向发生偏移，不再垂直于变换后的表面。
- 逆转置矩阵通过"反向补偿"缩放的各向异性，确保法线始终指向正确的垂直方向。

## 🛠 工程实践

- **引擎预计算法线矩阵**：主流引擎（Unity、Unreal）在 CPU 端预计算法线矩阵 $(\mathbf{M}_{3\times3}^{-1})^{\top}$，通过 Uniform Buffer 传递给 GPU。避免在着色器中实时计算矩阵求逆和转置，减少 GPU 计算开销。
- **静态物体 CPU 端预计算**：对于静态物体，模型矩阵在运行时不变，法线矩阵可以离线预计算并缓存。对于骨骼动画等动态物体，每帧需要重新计算法线矩阵，此时可以在顶点着色器或计算着色器中完成。
- **GPU 端精度注意事项**：在着色器中手动计算逆转置矩阵时，需要注意浮点精度。对于接近奇异的矩阵（极端缩放），求逆可能产生数值不稳定。可以设置缩放比例的上下限来避免退化情况。
- **非均匀缩放物体的必要处理**：当美术资源中存在非均匀缩放（如拉伸的柱体、压扁的球体）时，必须使用逆转置矩阵变换法线，否则光照计算会出现明显的明暗异常。
- **着色器中的实现方式**：在 GLSL 中，可以使用 `transpose(inverse(mat3(modelMatrix)))` 计算法线矩阵。在 HLSL 中对应为 `transpose((float3x3)modelMatrix).Inverse()` 或直接使用编译器内置函数。

## ⚠️ 踩坑经验

- **非均匀缩放物体光照异常排查**：某项目中，美术用 Scale 工具拉伸了一棵树的树干（X/Z 方向 1.0，Y 方向 2.0），导致树干表面出现奇怪的明暗条纹。排查发现是着色器中直接用 Model 矩阵变换法线，切换为逆转置矩阵后光照恢复正常。此类问题的特征是：只有非均匀缩放的物体受影响，均匀缩放物体正常。
- **移动端 GPU 矩阵求逆转置开销**：在移动端，着色器中实时计算 `inverse()` 和 `transpose()` 的开销较高（尤其在高 Draw Call 场景下）。建议在 CPU 端预计算法线矩阵并上传，或在美术规范中尽量避免非均匀缩放，从而允许直接用 Model 矩阵变换法线。
- **法线矩阵与 w 分量的陷阱**：法线是方向向量，变换时应该使用 `vec3(normalMatrix, n)` 而非 `vec4(n, 0.0) * modelMatrix`。如果错误地使用 `vec4(n, 1.0)`，平移分量会参与计算，导致法线完全错误。
- **骨骼动画中的累积误差**：在蒙皮动画中，每根骨骼的变换矩阵可能包含非均匀缩放，法线矩阵需要逐骨骼计算。如果骨骼层级很深，矩阵累积的浮点误差可能导致法线矩阵不准确，需要在动画烘焙阶段进行归一化和正交化处理。

## 🔗 延伸思考

- **切线空间的法线贴图是否受此影响？** 法线贴图存储的是切线空间中的法线偏移量，在采样后需要从切线空间变换到世界空间。这个变换使用的 TBN 矩阵（由切线、副切线、法线构成）同样受非均匀缩放影响。如果模型存在非均匀缩放，TBN 矩阵中的切线和副切线需要用模型矩阵变换，而法线分量需要用逆转置矩阵变换，否则法线贴图的效果也会失真。
- **Skinning 动画中的法线变换如何处理？** 在蒙皮动画中，顶点位置由多根骨骼加权混合：$\mathbf{p}' = \sum w_i \mathbf{M}_i \mathbf{p}$。但法线不能简单地加权混合（线性插值不保持单位长度和垂直性）。正确做法是对每个骨骼的法线矩阵做加权混合：$\mathbf{n}' = \text{normalize}(\sum w_i (\mathbf{M}_{i, 3\times3}^{-1})^{\top} \mathbf{n})$，或者使用 Dual Quaternion 方法来避免插值伪影。
- **逆转置矩阵与微分几何的关系**：从微分几何的角度看，法线是表面的伪向量（Pseudovector），其变换遵循余切空间（Cotangent Space）的变换规则，这正是逆转置矩阵的数学本质。理解这一点有助于在更复杂的几何处理（如曲面细分、位移映射）中正确处理法线变换。
