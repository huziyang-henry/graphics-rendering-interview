# Q01.12 什么是线性空间渲染？伽马校正在渲染管线中扮演什么角色？

**难度：** 🟢 初级
**所属章节：** [渲染管线基础](./index.md)

---

## 🎯 结论

- 线性空间渲染（Linear Space Rendering）是指在整个渲染管线的光照计算阶段，所有颜色值都在线性颜色空间（Linear Color Space）中进行运算，确保物理上正确的光照叠加和能量守恒。
- 伽马校正（Gamma Correction）是补偿显示器非线性响应的关键步骤：由于显示器对输入电压的响应近似幂函数（约 Gamma 2.2），如果不做校正，渲染结果会比预期更亮，暗部细节丢失，整体画面发灰。
- 现代渲染管线的标准流程是：输入纹理（Diffuse/Albedo）从 sRGB 转换为线性空间 → 在线性空间中进行所有光照计算 → 输出前做伽马 2.2 校正转回 sRGB 空间 → 显示器显示。

## 📐 原理解析

### 人眼感知与伽马编码

- 人眼对暗部变化的敏感度远高于亮部。在 8 位颜色通道（0-255）中，如果线性分配，暗部的相邻色阶差异人眼无法分辨，而亮部的色阶差异又过于明显。
- 伽马编码（Gamma Encoding）通过幂函数 $C_{\text{encoded}} = C_{\text{linear}}^{1/\gamma}$（通常 $\gamma = 2.2$）对颜色进行非线性映射，将更多的位深分配给暗部区域，从而更好地匹配人眼的感知特性。
- sRGB 颜色空间标准本质上就是伽马编码的一种规范化定义，其传输函数近似于 Gamma 2.2，但在极暗部有一个线性段以避免数值问题。sRGB 的精确转换公式为：

$$C_{\text{linear}} = \begin{cases} \frac{C_{\text{srgb}}}{12.92}, & C_{\text{srgb}} \leq 0.04045 \\ \left(\frac{C_{\text{srgb}} + 0.055}{1.055}\right)^{2.4}, & C_{\text{srgb}} > 0.04045 \end{cases}$$

### 显示器的非线性响应

- CRT 显示器（以及后续的 LCD/OLED）的电压-亮度响应近似幂函数：$L = V^{\gamma}$，其中 $\gamma \approx 2.2$。这意味着如果输入线性颜色值，显示器的输出亮度会偏暗。
- 伽马校正的目的是在最终输出前做逆变换：$C_{\text{output}} = C_{\text{linear}}^{1/2.2}$，使得经过显示器非线性响应后的最终亮度等于预期的线性亮度：$L = (C_{\text{linear}}^{1/2.2})^{2.2} = C_{\text{linear}}$。
- 历史上，伽马编码最初是为了补偿 CRT 显示器的非线性而设计的。但巧合的是，这种非线性编码恰好与人眼的感知特性匹配，因此即使现代显示器已经不再有 CRT 的非线性响应，伽马编码仍然被保留在图像标准和颜色空间中。

### 线性空间中光照计算的正确性

- 光照的物理叠加是线性操作：两个光源照射同一表面时，总辐射亮度是各自辐射亮度的和。这要求颜色值在线性空间中进行加法和乘法运算。
- 如果在 sRGB 空间（非线性空间）中直接进行光照计算，会导致以下问题：
  - **光照叠加过亮**：两个 sRGB 值相加不等于线性叠加。例如，两个 sRGB 值 0.5 叠加后在线性空间应为约 0.73，但在 sRGB 空间直接相加得到 1.0（过曝）。
  - **高光和反射计算错误**：PBR 中的 BRDF 计算基于物理公式，假设输入是线性辐射亮度。sRGB 输入会导致高光强度和衰减计算失真。
  - **颜色混合不准确**：纹理过滤（双线性、三线性）和 Alpha Blend 都隐含了颜色插值，这些操作在线性空间中才是物理正确的。

### 伽马校正的完整流程

- **输入阶段**：Diffuse/Albedo 纹理通常以 sRGB 格式存储，采样时需要硬件自动转换为线性值。这是通过在纹理加载时标记 sRGB 格式（OpenGL 的 `GL_SRGB8_ALPHA8`，DirectX 的 `DXGI_FORMAT_R8G8B8A8_UNORM_SRGB`）来实现的。
- **计算阶段**：所有光照计算（漫反射、镜面反射、环境光、阴影等）在线性空间中进行。
- **输出阶段**：最终颜色在写入帧缓冲前做伽马 2.2 校正：$C_{\text{output}} = C_{\text{linear}}^{1/2.2}$，或者使用 sRGB 格式的 RenderTarget，由硬件自动完成转换。

## 🛠 工程实践

- **sRGB 纹理的硬件自动转换**：现代 GPU 支持在纹理采样阶段自动执行 sRGB → Linear 转换。只需在创建纹理时指定 sRGB 格式，采样返回的值就是线性空间颜色。这一转换几乎零性能开销（硬件 LUT 查表实现）。
- **哪些纹理需要 sRGB 转换**：只有"颜色"类纹理需要 sRGB 转换，包括 Diffuse/Albedo、Emissive。以下纹理存储的是数据值而非颜色，**不需要** sRGB 转换：
  - 法线贴图（Normal Map）：存储的是方向向量
  - 金属度贴图（Metalness Map）：存储的是标量参数
  - 粗糙度贴图（Roughness Map）：存储的是标量参数
  - 高度图/位移贴图（Height/Displacement Map）：存储的是高度值
  - AO 贴图（Ambient Occlusion Map）：存储的是遮蔽系数
- **RenderTarget 的颜色空间选择**：光照计算的中间 RenderTarget 必须使用线性格式（如 `GL_RGBA16F`），不能使用 sRGB 格式，否则每次写入都会被错误地做伽马校正。只有最终输出的 Backbuffer 可以使用 sRGB 格式。
- **输出伽马校正的实现方式**：
  - 方式一：使用 sRGB 格式的 Backbuffer，硬件在最后一步自动做伽马校正（推荐）。
  - 方式二：在最终的后处理 Shader 中手动做 `pow(color, 1.0/2.2)`。
  - 方式三：使用 `glEnable(GL_FRAMEBUFFER_SRGB)`（OpenGL）全局启用 sRGB 写入转换。
- **HDR 渲染中的伽马校正**：在 HDR 渲染管线中，光照计算在线性空间的高动态范围 RenderTarget（如 `GL_RGBA16F`）上进行。Tone Mapping（色调映射）将 HDR 值压缩到 [0, 1] 范围后，再做伽马校正输出到 LDR 屏幕。

## ⚠️ 踩坑经验

- **未开启 sRGB 线性转换导致 PBR 效果异常**：某项目中，美术反馈 PBR 材质的高光不明显、整体画面偏灰偏暗。排查发现 Albedo 纹理没有标记为 sRGB 格式，导致采样得到的是 sRGB 空间的值而非线性值，光照计算在错误的颜色空间中进行。修复后高光和整体亮度恢复正常。这类问题的典型特征是：暗部细节丢失、金属材质缺乏高光反射、颜色整体偏灰。
- **后处理阶段的颜色空间混淆**：后处理效果（Bloom、DOF、Color Grading 等）需要在线性空间中操作。如果后处理在伽马校正之后的 sRGB 空间进行，会导致 Bloom 的光晕亮度不正确、Color Grading 的色阶映射失真。确保后处理链路的 RenderTarget 都是线性格式。
- **纹理工具导出设置**：美术工具（Substance Painter、Photoshop）导出纹理时需要注意颜色空间的设置。如果 Albedo 纹理被导出为线性空间（而非 sRGB），在 GPU 端又做了 sRGB → Linear 转换，会导致颜色被"双重校正"，整体偏亮偏灰。建议在纹理导入管线中统一处理颜色空间转换，减少美术手动配置的出错概率。
- **移动端硬件 sRGB 支持**：现代移动端 GPU（Mali、Adreno、Apple GPU）普遍支持硬件 sRGB 采样和写入转换，性能开销可忽略。但在较老的设备上（如 OpenGL ES 2.0 时代），可能需要手动在 Shader 中做伽马转换，此时需要注意性能影响。

## 🔗 延伸思考

- **伽马 2.2 的物理依据是什么？** 严格来说，伽马 2.2 并非来自纯物理推导，而是经验值。它的来源有两个方面：一是早期 CRT 显示器的电子枪响应特性（约 Gamma 2.2-2.5）；二是人眼的 Weber-Fechner 定律——主观感知与刺激强度的对数成正比。sRGB 标准中的传输函数（近似 Gamma 2.2 加上线性段）是这两个因素的综合折中。现代显示器的实际伽马值可能不同（如某些 OLED 屏幕约 2.1），但 sRGB 作为统一标准仍然被广泛使用。
- **HDR 显示器的 PQ/HLG 传递函数**：HDR 显示器使用不同的电光传递函数（EOTF）。PQ（Perceptual Quantizer，ST 2084）基于人眼的对比度敏感度函数，支持高达 10000 nits 的亮度范围；HLG（Hybrid Log-Gamma）则兼容 SDR 和 HDR 显示。在 HDR 渲染管线中，最终输出不再是简单的 Gamma 2.2 校正，而是需要根据目标显示器的 EOTF 选择对应的传递函数。这也是为什么现代引擎需要支持多种输出颜色空间的原因。
- **线性空间渲染与色彩管理的区别**：线性空间渲染解决的是光照计算的物理正确性问题，而色彩管理（Color Management）解决的是颜色在不同设备间的一致性问题。两者是正交的概念：线性空间渲染关注的是"计算对不对"，色彩管理关注的是"颜色准不准"。现代引擎（如 Unity 2019+ 的 Color Space 设置、Unreal 的 Color Management）通常将两者整合在一起，提供统一的颜色管线配置。
